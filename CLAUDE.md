工作方式（必须遵守）
0. 修改前必须输出
代码库现状分析（你看到了什么：目录结构/关键模块/可能相关文件）
本次目标（用1-3条可验证的句子描述）
修改计划（拆成小步骤，按顺序）
验收标准（Definition of Done）：必须是“可运行/可检查”的标准
1. 修改规则
优先最小改动：能小修不重构。
不要提交任何密钥、token、账号信息；不要把 .env 写进仓库。
每次改动后给出：改了哪些文件、为什么这么改、风险点。
2. 验证要求（改完必须执行并贴结果）
按顺序执行（把命令替换成项目真实命令）：
安装依赖：npm ci 或 npm install
代码检查：npm run lint
单测（如有）：npm test
构建：npm run build
若失败：
必须先贴关键错误日志（不要只说失败）
解释根因假设
给出最小修复方案
修复后再次跑上述命令，直到全部通过
3. 提交前审查清单（必须逐条自检）
变更是否只覆盖本次目标
是否引入未使用依赖/死代码
是否破坏类型/接口/路由
是否影响 Cloudflare Pages 构建输出目录
是否需要更新 README（功能/命令/部署/环境变量）
4. README 更新规则
如本次改动影响：
使用方式 / 配置 / 构建命令 / 环境变量 / 部署流程 则必须同步更新 README，并说明变更点。
5. 请完成这个任务，测试没通过，就别停止。
你可以自己看报错，自己写代码，自己重试。想试多少次就试多少次，只要没完成，就继续改。你每轮都能看到自己的上轮输出、错误日志或Git历史，然后再尝试修正问题，直至完成。
6. 需要用最轻量化的、具有鲁棒性的代码去实现，不影响项目的流畅及稳定性
7. 改好代码后，需要自己重新检查下是否完成了我们沟通的需求、要求，如果没完成，就重新更改代码，改到完成为止。
